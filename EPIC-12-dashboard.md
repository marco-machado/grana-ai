# EPIC-12: Dashboard Build (Umbrella)

**Status:** Not Started
**Dependencies:** EPIC-01, EPIC-02

## Overview

Umbrella epic that restructures the dashboard build into a phased approach. Replaces the fine-grained sequential dependency chain of EPICs 03–11 with a single epic containing 13 issues across 3 phases: foundation → vertical slices → Metabase.

Statement processing pipeline (EPIC-03) is intentionally excluded — it's ingestion infrastructure, not dashboard functionality. Categorization rules engine (EPIC-05) is also deferred — only Category CRUD is included here.

## Phase 1 — Foundation

### 12.1: Schema Extension — New Models & Transaction Fields

**Scope:** Add all remaining database models and extend the Transaction model with feature-linking fields. Single Prisma migration.

**Depends on:** EPIC-02 (Transaction model must exist)
**Reference:** EPIC-06, EPIC-07, EPIC-08, EPIC-09

#### New Enums

| Enum | Values |
|------|--------|
| `RecurringFrequency` | `WEEKLY`, `BIWEEKLY`, `MONTHLY`, `YEARLY` |
| `RecurringType` | `SUBSCRIPTION`, `BILL`, `INSTALLMENT` |
| `InstallmentFrequency` | `WEEKLY`, `BIWEEKLY`, `MONTHLY`, `YEARLY` |
| `BudgetStatus` | `ACTIVE`, `PAUSED` |

#### New Models

| Model | Key Fields |
|-------|-----------|
| `Budget` | id, category_id (FK), amount (Decimal(12,2)), period (Date, nullable — null = default template, first-of-month Date = monthly override), status (BudgetStatus, default ACTIVE), created_at, updated_at |
| `RecurringItem` | id, name, amount (Decimal(12,2)), frequency (RecurringFrequency), next_date (Date), type (RecurringType), category_id (FK → Category, nullable), account_id (FK → Account, nullable), created_at, updated_at |
| `InstallmentGroup` | id, description, total_amount (Decimal(12,2)), installments_total (Int), frequency (InstallmentFrequency), start_date (Date), category_id (FK → Category, nullable), account_id (FK → Account, nullable), created_at, updated_at. **`installments_paid` is computed** from `COUNT(transactions WHERE installment_group_id = group.id)`, not stored — consistent with SavingGoal's computed `current_amount`. |
| `SavingGoal` | id, name, target_amount (Decimal(12,2)), deadline (Date, nullable), linked_account_id (FK → Account, nullable), created_at, updated_at |

#### Transaction Extensions

Add 5 fields to the existing `Transaction` model:

| Field | Type | Notes |
|-------|------|-------|
| `is_recurring` | Boolean | Default false |
| `recurring_item_id` | UUID, nullable | FK → RecurringItem |
| `installment_group_id` | UUID, nullable | FK → InstallmentGroup |
| `is_savings_transfer` | Boolean | Default false |
| `saving_goal_id` | UUID, nullable | FK → SavingGoal |

#### Constraints

- `Budget`: unique on `(category_id, period)` — one budget per category per period. `period` is a Date (always first-of-month, e.g. `2026-03-01`); `null` = default template. Requires two constraints: (1) standard unique on `(category_id, period)` for non-null periods, (2) partial unique index on `category_id WHERE period IS NULL` to enforce one default template per category (PostgreSQL treats NULLs as distinct in unique constraints). **Migration note:** The partial unique index cannot be generated by `prisma migrate dev` — it requires a custom SQL block in the migration file (`CREATE UNIQUE INDEX ... WHERE period IS NULL`)
- `SavingGoal.linked_account_id` is optional — goals can exist without a dedicated savings account
- Transaction FKs to `RecurringItem`, `InstallmentGroup`, and `SavingGoal` use `onDelete: SetNull` — deleting a parent entity nullifies the FK on linked transactions

#### Files

- `app/prisma/schema.prisma` — add 4 enums, 4 models, extend Transaction
- Migration file (auto-generated by `prisma migrate dev`)

#### Acceptance Criteria

- [ ] Migration applies cleanly on top of EPIC-02's schema
- [ ] All 4 new models created with correct field types and defaults
- [ ] Transaction model has 5 new fields with proper FK relations
- [ ] `Budget` unique constraint on `(category_id, period)` enforced
- [ ] `npx prisma generate` produces client with all new types
- [ ] Existing tests still pass

## Phase 2 — Vertical Slices

Issues 12.2–12.8 are **parallel** for development — no build-time dependencies. Each delivers: Zod schema + API routes + UI page + tests + E2E tests. Note: 12.4 (Review) promotes staging → transactions, which 12.3 (List) reads — a runtime dependency, but both can be built independently.

**Conventions for all Phase 2 issues:**
- All list endpoints default to **50 per page**, max 100
- All pages use Next.js `loading.tsx` + `error.tsx` conventions for loading skeletons and error boundaries
- UI components use shadcn/ui primitives from `@/components/ui/`. Forms use `Dialog` for create/edit modals
- Reuse composite components from 12.1.5: DataTable (tables with sort/pagination) and CategorySelect (category combobox)
- **Toast notifications:** All mutations (create, update, delete, bulk actions) show a Sonner toast on success (green) and error (red). Success toasts auto-dismiss after 3s. Error toasts show the API error message and persist until dismissed.
- **Sidebar wiring:** Each issue must verify its page is reachable from the sidebar. The scaffold sidebar already includes entries for Transações, Orçamentos, Recorrentes, Parcelas, Metas, and Fontes. Issues 12.2 (Categories) and 12.4 (Review) add new entries — see their file lists. Issues 12.3, 12.5, 12.6, 12.7, 12.8 use existing sidebar entries — verify the `href` matches the route.
